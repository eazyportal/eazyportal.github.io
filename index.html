Introduction

eazyPortal is an implementation of a Modular Portal Framework. The main goal of this type of framework is to be modular in every possible way. This modularity is present in back-end and front-end as well.

Built on top of this modularity, a specific skeleton had been designed. It is called Behemoth. It can include any or all of these modules while it runs on a single application service. This inclusion is based on build-time configuration.

eazyPortal Structure

Every microservice (aka module) contains its UI, and they delivered as a Docker containers.

Every module is required to implement the Core module for back-end and Core-UI module for front-end. These Core modules contain common code, and encapsulate some required dependencies. All dependencies are required to be the latest ones. This applies for the Core and 3rd-party dependencies either.

The back-end is Java based and built on Spring and Spring Boot, while the front-end is built on React and React Library.

There is an eazyGradlePlugin Gradle plugin that is responsible to force all modules to look and built in the same way.

Contribution to the eazyPortal

Repository Structure

All eazyPortal repositories follow the Gitflow workflow. The master branch is used for releases only. Everything which goes to the master branch that will automatically release a new version. Each feature should go to the feature branch.

Those services which have UI component, the repository contains a submodule. These submodules follow the same Gitflow workflow principal. The purpose of this structure is to be able to decouple the front-end code and development from the back-end code and development. However, they have to be released together.

For further details about the required/predefined project structure, submodules, and their dependency tree check the eazyGradlePlugin.

How to Contribute

  1. Fork the repository where you want to contribute

  2. Clone your fork

  3. Name your fork as origin and add the forked one as upstream

  4. Implement the feature

  5. Send PR from your fork into the feature branch of the upstream repository

How to Create a Release

Merge everything into the feature branch. If there are UI changes what you want to release, merge the UI submodule into the feature branch as well. Use PRs for this purpose.

After everything is in order, use the Jenkins job (TBD) to release it. All artifacts will be published to Github packages (under development).

Build and Run eazyPortal

Unfortunately none of the artifacts published to maven artifactory, so it is required to build everything locally. (There is an initiative to host an artifactory in the future.)

There are multiple ways to run eazyPortal. For demonstration purposes, only some core modules will be listed here. However, all modules should work in the same way.

Fork all repositories, and use your fork to clone them. Add each of your fork as origin and these as upstream. To read more about the repository structure and how to contribute please read the contribution documentation.

Precondition

  • Properly configured Github account with ssh access to it. gradle.properties

  • Fork and clone the Layout module, the UPS module, and the Auth module.

    git clone --recurse-submodules git@github.com:<username>/eazyportal-layout.git layout
git clone --recurse-submodules git@github.com:<username>/eazyportal-user-permission-site.git ups
git clone --recurse-submodules git@github.com:<username>/eazyportal-auth.git auth

  • (Optional) To get the latest development version checkout to the feature branch for each module.

    git checkout --recurse-submodules feature

  • (Optional) If you want to test the latest UI features, it is required to checkout to the feature branch manually for the UI submodule.

    cd ui
git checkout feature
cd ..

  • Get the config from deploy config repository.

  • Install the Nginx locally, or use it from Docker container. For both cases use the configuration from the 'local' folder of the 'deploy config' repository.

    Start the Nginx as Docker container 
    dokcer-compose -f nginx.yaml up

  • Install the Redis locally or use it from Docker container.

    Start the Redis as Docker container 
    dokcer-compose -f redis.yaml up

Run eazyPortal from Intellij IDEA

After all of the preconditions are met:

  • Open each module separately with Intellij IDEA.

  • Use the preconfigured Run/Debug Configuration with - Nginx ending to start each application.

  • Open http://localhost and happy hacking.

Run eazyPortal from command line

After all of the preconditions are met:

  • Start each module with

    ./gradlew bootRun --args='--spring.profiles.active=nginx'

  • Open http://localhost and happy hacking.

Create a custom Module

This chapter will guide you through how to create a custom eazyPortal Module.

All official eazyPortal modules are use Kotlin DSL build script, so this tutorial will use that as well. Of course, if you wish to use Groovy you can do that.

Create the Module with Jenkins build

There will be a Jenkins job for this. TOOLS-8

Create the Module manually

Initialize a custom Module

  • Create an empty Gradle project (Project name: example)

  • Update the Gradle to the latest version

    ./gradlew wrapper --gradle-version=<gradle_version>

  • Apply the eazyGradlePlugin to the root project

    build.gradle.kts
    buildscript {
    repositories {
        gradlePluginPortal()
        maven {
            name = "github"
            url = uri("https://maven.pkg.github.com/eazyportal/eazyGradlePlugin")
            credentials(PasswordCredentials::class)
        }
    }
    dependencies {
        classpath("org.eazyportal.gradle:eazy-gradle-plugin:+")
    }
}

apply(plugin = "org.eazyportal.gradle.eazy-portal")

  • You can omit the plugins, group, repository, and dependencies statements, because all of them are provided by the plugin.

    Note
    		 The group will be set to "org.eazyportal.<rootProject.name >".

  • Add the following sub-module: api, service, and web

    settings.groovy.kts
    include("example-api")
project(":example-api").projectDir = file("api")

include("example-service")
project(":example-service").projectDir = file("service")

include("example-web")
project(":example-web").projectDir = file("web")
    Note
    		 Currently, these sub-modules are required, but it is planned to change this behaviour. (TOOLS-9)

  • Create a SpringBoot main class in the root project (src/main/java/)

    org.eazyportal.example.ExampleApplication.java
    @SpringBootApplication(
    scanBasePackages = {"org.eazyportal.core", "org.eazyportal.example"}
)
public class ExampleApplication {

    public static void main(String[] args) {
        SpringApplication.run(ExampleApplication.class, args);
    }

}
    Important
    		 It is required to add the Core module package to the scanBasePackages to enable eazyPortal preconfigurations.

Create REST controller

All official eazyPortal modules use the org.eazyportal.<service-name>.<module>.more.packages package structure, so this tutorial will use that as well.

  • Create the folder structure for the api module (api/src/main/java)

  • Create the response which extends the EazyResponse

    org.eazyportal.example.api.response.model.ExampleResponse.java

  • Create the folder structure for the web module (web/src/main/java)

  • Create the REST controller which extends the EazyRestController<BasicRequestContext>

    org.eazyportal.example.web.rest.contoller.ExampleContoller.java
    Note
    		 BasicRequestContext is a ModelAttribute that is created by the EazyRestController

  • Create the ExampleWebSecurityConfigurer which extends the EazyWebSecurityConfigurer. This is required to allow access to the new endpoint.

Customizing the UI content




Most of the official eazyPortal UI modules provides the ability to reuse the library in the Behemoth project.
To achieve this, it required to change the generated code.
The application should be as lightweight as possible, and should contain only the bare minimum.
Therefore, the library should contain everything.




Refactor the generated code




    

  • 

    Move App*.js and index.css from the application (example/src/) to the library (src/).
    

    • 

  • 

    Replace the content of the index.js of the application with this:
    

    

    example/src/ExampleComponent.js
    

    

    import Example from '@eazyportal/example-ui'

import React from 'react'
import ReactDOM from 'react-dom'

import '@eazyportal/example-ui/dist/index.css'

ReactDOM.render(<Example.App />, document.getElementById('root'))
    

    

    

    • 

  • 

    Move and rename the index*.js of the library to src/components/ExampleComponent*.js.
Move and rename the styles.module.css to src/components/ExampleComponent.module.css.
It is required to change the styles.module.css in the ExampleComponent.js too to point to the correct css file.
    

    • 

  • 

    Update the import of ExampleComponent and index.css in the App.js to point to the renamed files.
    

    

    src/App.js

Display response from REST endpoint




After refactoring the generated code, it is ready to be customized.
During the previous chapter 2 REST endpoint was implemented, so let’s use them.






    

  • 

    Create a new component ExampleListComponent
    

    
src/components/ExampleListComponent.js

    

    

    

    import Core from '@eazyportal/core-ui'

import React from 'react'

import { ExampleComponent } from './ExampleComponent'

class ExampleListComponent extends React.Component {

  state = {
    examples: null
  }

  componentDidMount() {
    Core.axiosService.get(`/api/examples`)
      .then(response => this.setState({examples: response.data}))
      .catch(error => console.log(error))
  }

  render() {
    let renderedExamples = null
    if (this.state.examples) {
      renderedExamples = Object.entries(this.state.examples)
        .map(([index, example]) => {
          return <ExampleComponent
            key={index}
            text={`Hello, my name is ${example.name} and I am ${example.age} years old.`}
          />
        })
    }

    return (
      <React.Fragment>
        {renderedExamples}
      </React.Fragment>
    )
  }

}

export default ExampleListComponent
    

    

    

    

    

    • 

  • 

    Update the App component to use the new component
    

    
src/App.js

    

    

    

    import React from 'react'

import { ExampleComponent } from './components/ExampleComponent'
import ExampleListComponent from './components/ExampleListComponent'

import source from './App.module.css'

class App extends React.Component {

  state = {
    loadData: false
  }

  buttonClickHandler = () => {
    this.setState({loadData: !this.state.loadData})
  }

  render() {
    let renderedComponent
    if (this.state.loadData) {
      renderedComponent = <ExampleListComponent />
    }
    else {
      renderedComponent = <ExampleComponent text="Create React Library Example 😄" />
    }

    return (
      <div>
        <div className={source.sourceSwitchWrapper}>
          <button className={source.sourceSwitchButton} onClick={this.buttonClickHandler}>Switch between data sources</button>
        </div>
        <div>
          {renderedComponent}
        </div>
      </div>
    );
  }
}

export default App

Continuous UI development




For continuous development this approach is not the best, because it requires you to constantly stop, rebuild, and restart all applications.
To avoid this, fortunately there is an option.
You have to start everything separately.






Unfortunately this solution requires a proxy (e.g. Nginx), or some hardcoded values.
You have to proxy the REST requests to the Spring Boot application, OR enable @CrossOrigin on the endpoint and provide the full URL for axios.
(At least in the current version.)






Start the Spring Boot application from the projects main folder.








$ ./gradlew bootRun -x deployUi








Start "watching" for library changes from the ui module folder.








$ npm run start








Start the UI application from the example folder of the ui module.








$ npm run start
