Introduction

eazyPortal is an implementation of a Modular Portal Framework. The main goal of this type of framework is to be modular in every possible way. This modularity is present in back-end and front-end as well.

Built on top of this modularity, a specific skeleton had been designed. It is called Behemoth. It can include any or all of these modules while it runs on a single application service. This inclusion is based on build-time configuration.

eazyPortal Structure

Every microservice (aka module) contains its UI, and they delivered as a Docker containers.

Every module is required to implement the Core module for back-end and Core-UI module for front-end. These Core modules contain common code, and encapsulate some required dependencies. All dependencies are required to be the latest ones. This applies for the Core and 3rd-party dependencies either.

The back-end is Java based and built on Spring and Spring Boot, while the front-end is built on React and React Library.

There is an eazyGradlePlugin Gradle plugin that is responsible to force all modules to look and built in the same way.

Getting Started With EazyPortal

Build and Run eazyPortal

Unfortunately none of the artifacts published to maven artifactory, so it is required to build everything locally. (There is an initiative to host an artifactory in the future.)

There are multiple ways to run eazyPortal. For demonstration purposes, only some core modules will be listed here. However, all modules should work in the same way.

Fork all repositories, and use your fork to clone them. Add each of your fork as origin and these as upstream. To read more about the repository structure and how to contribute please read the contribution documentation.

Precondition

  • Properly configured Github account with ssh access to it. gradle.properties

  • Fork and clone the Layout module, the UPS module, and the Auth module.

    git clone --recurse-submodules git@github.com:<username>/eazyportal-layout.git layout
git clone --recurse-submodules git@github.com:<username>/eazyportal-user-permission-site.git ups
git clone --recurse-submodules git@github.com:<username>/eazyportal-auth.git auth

  • (Optional) To get the latest development version checkout to the feature branch for each module.

    git checkout --recurse-submodules feature

  • (Optional) If you want to test the latest UI features, it is required to checkout to the feature branch manually for the UI submodule.

    cd ui
git checkout feature
cd ..

  • Get the config from deploy config repository.

  • Install the Nginx locally, or use it from Docker container. For both cases use the configuration from the 'local' folder of the 'deploy config' repository.

    Start the Nginx as Docker container 
    dokcer-compose -f nginx.yaml up

  • Install the Redis locally or use it from Docker container.

    Start the Redis as Docker container 
    dokcer-compose -f redis.yaml up

Run eazyPortal from Intellij IDEA

After all of the preconditions are met:

  • Open each module separately with Intellij IDEA.

  • Use the preconfigured Run/Debug Configuration with - Nginx ending to start each application.

  • Open http://localhost and happy hacking.

Run eazyPortal from command line

After all of the preconditions are met:

  • Start each module with

    ./gradlew bootRun --args='--spring.profiles.active=nginx'

  • Open http://localhost and happy hacking.

Create the Service Manually

All official eazyPortal service use the org.eazyportal.<service-name>.<module>.more.packages package structure, so this tutorial will use that as well.

This example describes the necessary steps to implement a complete eazyPortal service. Can create and retrieve data via REST endpoints, and persists it into a DB.

Initialize the Project

  • Create an empty Gradle project with the name of example

  • Update the Gradle Wrapper to the latest version

    ./gradlew wrapper --gradle-version=<gradle_version>

  • Configure the project to use GitHub for plugin artifactory

    settings.gradle.kts
    pluginManagement {
    repositories {
        maven {
            name = "github"
            url = uri("${extra["githubUrl"] as String}/*")

            credentials {
                password = extra["githubPassword"] as String
                username = extra["githubUsername"] as String
            }
        }
    }
}

  • Apply the org.eazyportal.plugin.portal to the root project

    settings.gradle.kts
    pluginManagement {
    plugins {
        id("org.eazyportal.plugin.portal") version(extra["eazyPortalPluginVersion"] as String)
    }
}
    build.gradle.kts
    plugins {
    id("org.eazyportal.plugin.portal")
}
    gradle.properties
    eazyPortalPluginVersion = +

  • You can omit the plugins, group, repository, and dependencies statements, because all of them are provided by the plugin.

    Note
    		 The group will be set to "org.eazyportal.<rootProject.name >".
Implement the main aka application module

  • Create a SpringBoot main class in the root project (src/main/java/)

    org.eazyportal.example.ExampleApplication.java (see it on GitHub)
    @SpringBootApplication(scanBasePackages = {
    "org.eazyportal.core",
    "org.eazyportal.example"
})

    public static void main(String... args) {
        SpringApplication.run(ExampleApplication.class, args);
    }

}

  • Enable some locally running service configuration (enable: Swagger, CORS)

    application.yaml
    spring:
  profiles:
    active: local
    Important
    		 It is required to add the Core module package to the scanBasePackages to enable eazyPortal preconfigurations.
    Note
    		 Currently, eazyPortal supports only Java, but the Kotlin support is coming.

Implement the dao module

  • Configure the submodule

    settings.groovy.kts
    include("dao")
project(":dao").name = "example-dao"

  • Create the dao/src/main/java folder structure

  • Create a DB representation entity in the dao module which implements the EazyEntity

    org.eazyportal.example.dao.model.ExampleEntity.java (see it on GitHub)
    @Entity
public class ExampleEntity implements EazyEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(nullable = false, unique = true, updatable = false)
    private long id;

    @Column
    @Nullable
    private Integer age;

    @Column(nullable = false)
    private String name;

    // Getters

    public static class Builder implements EazyEntity.Builder {

        // Builder

    }

}

  • Create a Spring-Data @Repository to establish the connection with the DB

    org.eazyportal.example.dao.repository.ExampleRepository.java (see it on GitHub)
    @Repository
public interface ExampleRepository extends CrudRepository<ExampleEntity, Long> {

}

  • Create a DAO class with create and getById methods, and annotate it with @EazyDao (see it on GitHub)

    org.eazyportal.example.dao.ExampleDao.java
    @EazyDao
public class ExampleDao {

    @Autowired
    private ExampleRepository exampleRepository;

    public ExampleEntity create(ExampleEntity entity) {
        return exampleRepository.save(entity);
    }

    public Stream<ExampleEntity> getAll() {
        return StreamSupport.stream(exampleRepository.findAll().spliterator(), false);
    }

    public ExampleEntity getById(long id) {
        return exampleRepository.findById(id)
                .orElseThrow(() -> new EntityNotFoundException("Example entity does not found with id:" + id));
    }

}
Implement the service module

  • Configure the submodule

    settings.groovy.kts
    include("service")
project(":service").name = "example-service"

  • Create the service/src/main/java folder structure

  • Create a mapper class wich implements EazyEntityMapper

    org.eazyportal.example.service.mapper.ExampleMapper.java (see it on GitHub)
    @Component
public class ExampleEntityMapper implements EazyEntityMapper<ExampleEntity, ExampleResponse> {

    @Override
    public ExampleResponse map(ExampleEntity entity) {
        return exampleResponseBuilder()
            .withAge(entity.getAge())
            .withName(entity.getName())
            .build();
    }

}

  • Create a service class

    org.eazyportal.example.service.ExampleService.java (see it on GitHub)
    @Component
public class ExampleService {

    @Autowired
    private ExampleDao exampleDao;
    @Autowired
    private ExampleEntiryMapper exampleEntiryMapper;

    public ExampleResponse create(CreateExampleRequest createExampleRequest) {
        var exampleEntity = exampleEntityBuilder()
            .build(exampleDao::create);

        return exampleEntiryMapper.map(exampleEntity);
    }

    public List<ExampleResponse> getAll() {
        return exampleDao.getAll()
            .map(exampleEntityMapper::map)
            .collect(Collectors.toList());
    }

    public ExampleResponse getById(long id) {
        return exampleEntiryMapper.map(exampleDao.getById(id));
    }

}

The reason is behind that the service layer returns with the response is simple. The web layer does the verification on the request and just act as a proxy in case of the response. And the behemoth module is depending on the service layer, but you will hear more about that later.

Implement the web module

Every REST controller should implement the EazyRestController and have only one purpose to exist

  • Configure the submodule

    settings.groovy.kts
    include("web")
project(":web").name = "example-web"

  • Create the web/src/main/java folder structure

  • Create a REST controller which implements the EazyRestController for creating new Examples

    org.eazyportal.example.web.rest.contoller.CreateExampleRestController.java (see it on GitHub)
    @RestController
@RequestMapping("/api/example")
public class CreateExampleRestController implements EazyRestController {

    @Autowired
    private ExampleService exampleService;

    @PostMapping
    public ExampleResponse create(@RequestBody CreateExampleRequest createExampleRequest) {
        return exampleService.create(createExampleRequest);
    }

}

  • Create a REST controller which implements the EazyRestController for getting Examples

    org.eazyportal.example.web.rest.contoller.GetExampleRestController.java (see it on GitHub)
    @RestController
@RequestMapping("/api/example")
public class GetExampleRestController implements EazyRestController {

    @Autowired
    private ExampleService exampleService;

    @GetMapping
    public List<ExampleResponse> getAll() {
        return exampleService.getAll();
    }

    @GetMapping("/{id}")
    public ExampleResponse getById(@PathVariable long id) {
        return exampleService.getById(id);
    }

}

  • Optionally, if you want to enable public access to any endpoint, you have to configure it. Create a security configurer which implements the EazyWebSecurityConfigurer.

    org.eazyportal.example.web.security.config.ExampleWebSecurityConfigurer.java (see it on GitHub)
    @Component
public class ExampleWebSecurityConfigurer implements EazyWebSecurityConfigurer {

    @Override
    public void configure(ExpressionUrlAuthorizationConfigurer<HttpSecurity>.ExpressionInterceptUrlRegistry authorizeRequests) {
        authorizeRequests
                .antMatchers(HttpMethod.GET, "/api/examples/**").permitAll();
    }

}

  • Optionally, if you want to enable swagger, you can configure it

    application.yaml
    springdoc:
  swagger-ui:
    enabled: true
Create the UI Library and Application

All official eazyPortal UI modules are written in React. It has 2 parts a library and an application. The library is reused in the Behemoth project. Therefore, the application should be as lightweight as possible, and should contain only the bare minimum, so the library should contain everything.

UI modules are stored in Git submodules. So they can have their limited but separated lifecycles as micro-frontends. They have limited lifecycles because they are released together with the applications.

  • Create a new git repository: example-ui

  • Add an initial commit and push it to GitHub

    Important
    		 Adding an empty repository as a submodule will fail. This is why an initial commit is required in that repository.

  • Add the repository to the project as a Git submodule with the name of ui

    $ git submodule add git@github.com:eazyportal/example-ui.git ui

  • Configure the submodule

    settings.groovy.kts

  • The backend returns with multiple example entities, so create a wrapper for them ExampleListComponent

    src/components/ExampleListComponent.js
    import { axiosService } from '@eazyportal/core-ui-lib'

import React from 'react'

import ExampleComponent from './ExampleComponent'

class ExampleListComponent extends React.Component {

  state = {
    examples: null
  }

  componentDidMount() {
    axiosService.get(`/api/examples`)
      .then(response => this.setState({examples: response.data}))
      .catch(error => console.log(error))
  }

  render() {
    let renderedExamples = null
    if (this.state.examples?.length) {
      renderedExamples = Object.entries(this.state.examples)
        .map(([index, example]) => {
          return <ExampleComponent
            key={index}
            text={`Hello, my name is ${example.name} and I am ${example.age} years old.`}
          />
        })
    }
    else {
      renderedExamples = <ExampleComponent text="Database is empty." />
    }

    return (
      <React.Fragment>
        {renderedExamples}
      </React.Fragment>
    )
  }

}

export default ExampleListComponent

  • Create the main entry point of the library

    src/App.jsx
    import React from 'react'

import ExampleComponent from './components/ExampleComponent'
import ExampleListComponent from './components/ExampleListComponent'

import source from './App.module.css'

class App extends React.Component {

  state = {
    loadData: false
  }

  buttonClickHandler = () => {
    this.setState({
      loadData: !this.state.loadData
    })
  }

  render() {
    let renderedComponent
    if (this.state.loadData) {
      renderedComponent = <ExampleListComponent />
    }
    else {
      renderedComponent = <ExampleComponent text="Create React Library Example 😄" />
    }

    return (
      <div>
        <div className={source.loadDataWrapper}>
          <button className={source.loadDataButton} onClick={this.buttonClickHandler}>Load data from backend</button>
        </div>
        <div>
          {renderedComponent}
        </div>
      </div>
    );
  }
}

export default App

Continuous UI development

For continuous development this approach is not the best, because it requires you to constantly stop, rebuild, and restart all applications. To avoid this, fortunately there is an option. You have to start the frontend and the backend separately (as stated before).

Unfortunately, this solution requires a proxy (e.g. Nginx), or some hardcoded values. You have to proxy the REST requests to the Spring Boot application, OR provide the full URL for axios. (At least in the current version.)

Contribution to the eazyPortal

Repository Structure

All eazyPortal repositories follow the Gitflow workflow. The master branch is used for releases only. Everything which goes to the master branch that will automatically release a new version. Each feature should go to the feature branch.

Those services which have UI component, the repository contains a submodule. These submodules follow the same Gitflow workflow principal. The purpose of this structure is to be able to decouple the front-end code and development from the back-end code and development. However, they have to be released together.

For further details about the required/predefined project structure, submodules, and their dependency tree check the eazyGradlePlugin.

How to Contribute

  1. Fork the repository where you want to contribute

  2. Clone your fork

  3. Name your fork as origin and add the forked one as upstream

  4. Implement the feature

  5. Send PR from your fork into the feature branch of the upstream repository

How to Create a Release

Merge everything into the feature branch. If there are UI changes what you want to release, merge the UI submodule into the feature branch as well. Use PRs for this purpose.

After everything is in order, use the Jenkins job (TBD) to release it. All artifacts will be published to Github packages (under development).
